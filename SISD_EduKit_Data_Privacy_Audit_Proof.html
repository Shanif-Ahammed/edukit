Executive Summary

The SISD EduKit - Teacher Portal includes an interactive helper chatbot (AiAssistant) to assist teachers in exporting templates, setting assessment targets, and troubleshooting file headers.

To satisfy Dubai **DSIB Pupil Protection Guidelines** and the **UAE Personal Data Protection Law (PDPL)**, the portal enforces absolute data isolation. Under this system architecture, **student rosters, grades, or names are structurally and programmatically prevented from being accessed by, transmitted to, or utilized for training any AI models.**

This document provides the concrete, verifiable code implementations proving that student records reside exclusively in isolated local browser threads, and that chatbot interactions carry zero exposure risk.

1. 100% Local Roster Ingestion (No Remote Uploads)

When a teacher imports a grading sheet or roster connected from iSAMS, the file is never uploaded to any remote server or backend database. It is parsed completely inside the browser utilizing client-side standard libraries.

Verification Code: Dashboard Ingestion (Dashboard.jsx)

Below is the official React implementation in src/tools/Dashboard.jsx. Note that standard local browser structures (FileReader) and the local library SheetJS (XLSX.read) handle the entire parse lifecycle: 

const handleFile = (file) => {
  if (!file) return;
  const name = file.name;
  const ext = name.split('.').pop().toLowerCase();
  
  // 1. Restrict processing strictly to spreadsheets
  if (ext !== 'xlsx' && ext !== 'xls') return;

  const reader = new FileReader(); // Browser-native file API
  reader.onload = (e) => {
    try {
      const data = e.target.result;
      
      // 2. Parse binary excel stream locally in the client-side sandbox
      const workbook = XLSX.read(data, { type: 'binary' });
      const sheetName = workbook.SheetNames[0];
      const sheet = workbook.Sheets[sheetName];
      
      // 3. Extracted rows mapped straight to local transient state
      const rows = XLSX.utils.sheet_to_json(sheet);
      
      // 4. Connect directly to React local context (No network hooks engaged)
      connectData(name, rows);
    } catch (err) {
      console.error("Local SheetJS extraction error:", err);
    }
  };
  reader.readAsBinaryString(file); // Triggers local parser read
};

Audit Conclusion: Since there are no database adapters, Axios posts, or remote fetch calls triggered in the file reader onload event, roster information remains locked within the physical browser's runtime heap.

2. Programmatic Chatbot Isolation from Student Records

The AI Assistant component (AiAssistant.jsx) is structurally isolated from active grades. It does not import or implement React Context Hooks connecting it to student records (such as useData), making student profiles physically invisible to the chatbot file boundaries.

Verification Code: Request Serialization Payload (AiAssistant.jsx)

The active API fetch call in src/tools/AiAssistant.jsx compiles only the user-typed support thread and a static guide instruction context. The student roster objects are structurally omitted from the request body: 

const handleSend = async (textToSend) => {
  const queryText = textToSend.trim();
  if (!queryText) return;

  try {
    // 1. Map ONLY chat thread bubbles (Sender and typed prompt message)
    const nativeContents = messages.map(m => ({
      role: m.sender === 'user' ? 'user' : 'model',
      parts: [{ text: m.text }]
    }));

    // 2. Push active chat query input
    nativeContents.push({
      role: 'user',
      parts: [{ text: queryText }]
    });

    // 3. Feed the generic user-manual system instruction rules
    const nativeSystemInstruction = {
      parts: [{ text: SYSTEM_INSTRUCTION }]
    };

    // 4. POST payload strictly containing text fields (No student rosters)
    const response = await fetch('/api/gemini', {
      method: 'POST',
      headers: { 'Content-Type': 'application/json' },
      body: JSON.stringify({
        contents: nativeContents,             // Interactive text lines
        systemInstruction: nativeSystemInstruction // User guide rules
      })
    });
    
    const data = await response.json();
    setMessages(prev => [...prev, { sender: 'ai', text: data.candidates[0].content.parts[0].text }]);
  } catch (err) {
    console.error(err);
  }
};

Audit Conclusion: The student array (students) is never declared, imported, or serialized into the outgoing request payload. No student name, grade, or classroom parameter is transited over the internet during help assistant queries.

3. Stateless Serverless API Proxy Architecture

To prevent frontend API key exposure, requests route through a serverless cloud proxy hosted on Vercel (api/gemini.js). The function acts strictly as an anonymous gateway, hiding credential keys, while writing no files and storing no histories.

Verification Code: Serverless Proxy Inbound Route (api/gemini.js)

The node.js serverless route receives prompt variables from the client and proxies them securely to the Google API endpoint, storing absolutely nothing: 

export default async function handler(req, res) {
  // Allow CORS headers
  res.setHeader('Access-Control-Allow-Origin', '*');
  if (req.method === 'OPTIONS') return res.status(200).end();
  if (req.method !== 'POST') return res.status(405).end();

  const apiKey = process.env.GEMINI_API_KEY;
  const { contents, systemInstruction } = req.body;

  try {
    // Securely proxy query to Google Developer endpoints
    const response = await fetch("https://generativelanguage.googleapis.com/v1beta/models/gemini-3.1-flash-lite:generateContent", {
      method: "POST",
      headers: {
        "Content-Type": "application/json",
        "X-goog-api-key": apiKey
      },
      body: JSON.stringify({ contents, systemInstruction })
    });

    const data = await response.json();
    // Returns data cleanly; zero database logging or file writing
    return res.status(200).json(data);
  } catch (error) {
    return res.status(500).json({ error: 'Internal Gateway Error' });
  }
}

4. Google Developer API Data Governance Terms

The serverless proxy routes requests exclusively through **Google's Developer API Tier** (Generative Language API), which is bound by professional developer services agreements.

Unlike consumer-facing chatbot interfaces, Google's developer API tier explicitly protects developer inputs:

  • Strict Non-Training Policy: Prompts, input chat tokens, and structured system instructions sent through developer API keys are strictly excluded from model training cycles. They are never saved, logged, or processed to train Google's models.
  • Transient Operations: Prompts are treated as transient query assets, processed strictly to compile the immediate completion response and purged from server memory buffers in accordance with Google's cloud compliance frameworks.
  • No Repository Exposure: Google has no access to the application source code files, configurations, or localized databases hosted on the client device.
